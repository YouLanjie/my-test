构建系统要求
============
1. 自查找(src/**/\(.*\).c -> .build/{??.o,lib??.a} -> bin/\1)
  1. 简单独立.c 编译成 .o 生成单独的elf文件
  2. 具有依赖/复用 .c 编译成 .o 自动一起连接
  3. 经常被复用的 .c 编译成 .o 后先合并成 libxx.a 后再连接
  4. 需要链接库(-lm,-lasound,-lncuses,-lSDL2,-liconv,-lavformat -lavcodec -lavutil -lswresample)的自动链接
  5. 能配置特殊编译选项（-fopenmp）
  6. 支持通过gcc -DXXX取消某些库依赖的在检测到无法链接对应库后自动加上对应-DXXX选项
  7. 缺乏库依赖而编译/链接终止的不导致其他独立程序的编译终止
2. 自依赖(.h -> .c -> .o/.a -> elf)
3. 轻量级
4. （最好）跨平台
5. 易于配置（重复工作少）

已有方案
============
1. bash脚本
  - 配置简单(switch...case...)
  - 相对轻量
  - 依赖关系暂未实现
  - 并行处理难
  - 配置硬编码分布在脚本各处
  - 字符串处理依赖频繁启动子进程,系统调用多,速度缓慢
2. Makefile
  - 轻量
  - 现成依赖处理
  - gcc -M 头文件依赖支持
  - 配置麻烦
  - 多层级.c结构到扁平化elf输出依赖关系构建困难
3. 自制build.c
  - 仅直接依赖编译器
  - 趣味性强
  - 系统调用有限速度快
  - 专一性强
  - 需要搓大量底层轮子（字符串拼接、路径处理）
  - 配置不一定简单

潜在方案
============
1. 自制build.py
  - 语法比bash优秀
  - 语法简单，易于配置（理论上）
  - 单进程完成计算无须子进程
  - 依赖过于重量级(虽然说好像python本体比gcc小就是了)
2. cmake
  - 跨平台？
  - 隐式变量巨多，配置困难
  - 配置不一定简单
  - 生成的Makefile过于臃肿运行缓慢需要换ninja
3. Meson
  - 依赖Python，过于重量级
